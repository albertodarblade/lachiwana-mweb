# Implementation Plan: Sliding Session Auth — Frontend Integration

**Branch**: `015-sliding-session-auth` | **Date**: 2026-05-25 | **Spec**: [spec.md](./spec.md)  
**Input**: Feature specification from `specs/015-sliding-session-auth/spec.md`

## Summary

Migrate the frontend authentication layer from a localStorage-persisted JWT to a sliding session model: Access Tokens are stored exclusively in memory, the Refresh Token lives in an HttpOnly cookie managed by the backend, and a single-flight 401 interceptor in the API client silently rotates the token without user interruption. A startup restore check runs on page load before Framework7 renders, showing a loading spinner while the silent refresh completes.

The backend implementation is complete (`../Lachiwana-service` branch `010-sliding-session-auth`). This plan covers only the frontend.

## Technical Context

**Language/Version**: JavaScript (ES2022) — Vite + React 19  
**Primary Dependencies**: Framework7 v9 (UI + routing), TanStack Query v5 (data), pnpm (package manager)  
**Storage**: In-memory module variable (token), localStorage (user profile only), HttpOnly cookie (Refresh Token, managed by backend)  
**Testing**: Manual (per Constitution IX)  
**Target Platform**: Mobile PWA (iOS Safari, Android Chrome); desktop secondary  
**Project Type**: Mobile web application (SPA)  
**Performance Goals**: Refresh round-trip invisible to user (< 1s); proactive renewal fires 5 min before expiry  
**Constraints**: Token must never touch localStorage/sessionStorage; refresh must de-duplicate concurrent calls  
**Scale/Scope**: Single-user session per tab; multiple sessions per user across devices (handled by backend)

## Implementation Details

### 1. `src/stores/tokenStore.js` (NEW)

Module-level variables:
- `_token` — current Access Token string or null
- `_expiresAt` — ISO timestamp string or null
- `_renewalTimer` — setTimeout ID or null

Functions:
- `getToken()` — returns `_token`
- `setToken(accessToken, expiresAt)` — stores values, cancels existing timer, calls `_scheduleRenewal(expiresAt)`
- `clearToken()` — sets both to null, calls `clearTimeout(_renewalTimer)`

Private `_scheduleRenewal(expiresAt)`:
- Parses `expiresAt` to ms
- Computes `delay = expiresAt - Date.now() - 5_minutes`
- If `delay <= 0`, call `_proactiveRefresh()` immediately
- Else `_renewalTimer = setTimeout(_proactiveRefresh, delay)`

Private `_proactiveRefresh()`:
- Imports `refreshToken` from `src/api/auth.js`
- On success: calls `setToken(accessToken, expiresAt)`
- On 401: calls `clearToken()`, `window.location.replace('/login?expired=1')`
- On 429: no-op (user is active; reactive refresh will handle the next 401)
- Console log: `[auth] proactive refresh triggered`, `[auth] proactive refresh success/failed`

### 2. `src/api/auth.js` (NEW)

Raw `fetch` calls (NOT through `client.js` to avoid circular dependency):

```javascript
const BASE = import.meta.env.LACHIWANA_SERVICE_URL + '/api/v1/auth'

export async function refreshToken() {
  const res = await fetch(`${BASE}/refresh`, {
    method: 'POST',
    credentials: 'include',
  })
  if (!res.ok) {
    const err = new Error('refresh_failed')
    err.status = res.status
    throw err
  }
  const { data } = await res.json()
  return data  // { accessToken, expiresIn, expiresAt }
}

export async function signOut() {
  await fetch(`${BASE}/signout`, {
    method: 'POST',
    credentials: 'include',
  })
}
```

Console log: `[auth] refresh success`, `[auth] signout called`

### 4. `src/api/client.js` (MODIFIED)

Consolidate `request`, `jsonRequest`, `getBlob`, `postForm` into a single internal `_request(method, path, options)` dispatcher. This eliminates ~80 lines of duplicated 401 handling.

Module-level state:
- `let _refreshPromise = null`

`_request` flow:
1. Build headers: `Authorization: Bearer ${getToken()}` if token exists
2. Call `fetch(...)`
3. If `response.status === 401`:
   a. If `_refreshPromise` is null: `_refreshPromise = refreshToken().finally(() => _refreshPromise = null)`
   b. Await `_refreshPromise`
   c. On success: `setToken(data.accessToken, data.expiresAt)`; retry original request ONCE
   d. On error with `status === 401`: `clearSession()`; `queryClient.clear()`; `window.location.replace('/login?expired=1')`; throw
   e. On error with `status === 429`: show F7 toast `'Demasiadas solicitudes, espera un momento.'`; throw
4. If not ok: throw `new Error('HTTP {status}: {statusText}')`
5. Return response (json / blob as needed)

Imports added: `getToken`, `setToken` from `tokenStore`; `refreshToken` from `api/auth`; `f7` from `framework7-react`

Replace `getSession`, `clearSession` calls with `getToken`, `setToken`, `clearToken` and `clearSession` (for full sign-out path).

Console log: `[auth] 401 intercepted, refreshing token`, `[auth] token refreshed, retrying`, `[auth] refresh failed, signing out`, `[auth] rate limited on refresh`

### 5. `src/pages/AuthCallbackPage.jsx` (MODIFIED)

URL params to read: `accessToken`, `expiresIn`, `expiresAt`, `user`, `error` (replacing `token`, `user`)

**Popup flow** (when `localStorage.getItem('lachiwana_oauth_popup') === '1'`):
1. Read params
2. If `error` or missing `accessToken`/`user`: `finish('/login?error=auth_failed')`
3. Parse `user`, call `authStore.setUser(user)`
4. Write `JSON.stringify({ accessToken, expiresAt })` to `localStorage.setItem('lachiwana_oauth_token', ...)`
5. Remove `lachiwana_oauth_popup`, set `lachiwana_oauth_done` to destination
6. `window.close()`

**Redirect flow** (no popup flag):
1. Read params, validate
2. Parse `user`, call `authStore.setUser(user)`
3. Call `tokenStore.setToken(accessToken, expiresAt)`
4. Clear URL params: `window.history.replaceState({}, '', destination)` 
5. `window.location.replace(destination)`

Console log: `[auth] token captured at callback`

### 8. `src/App.jsx` (MODIFIED)

Add session initialization:

```javascript
const [sessionChecked, setSessionChecked] = useState(false)

useEffect(() => {
  const path = window.location.pathname
  // Auth callback handles its own token setup; login doesn't need a session
  if (path === '/auth/callback' || path.startsWith('/login')) {
    setSessionChecked(true)
    return
  }
  if (getToken()) {
    // Token already present (shouldn't happen on cold start, defensive)
    setSessionChecked(true)
    return
  }
  refreshToken()
    .then(({ accessToken, expiresAt }) => {
      setToken(accessToken, expiresAt)
    })
    .catch(() => {
      // No session to restore — ProtectedRoute will redirect to login
    })
    .finally(() => {
      setSessionChecked(true)
    })
}, [])
```

Render:
```jsx
if (!sessionChecked) {

      data-testid="app-session-loading"
      style={{ display: 'flex', alignItems: 'center', justifyContent: 'center', minHeight: '100vh' }}

      <Preloader size={44} />
    </div>
  )
}
```

## Complexity Tracking

| Violation | Why Needed | Simpler Alternative Rejected Because |
|-----------|------------|--------------------------------------|
| **Constitution V** — `POST /api/v1/auth/refresh` uses raw `fetch` outside TanStack Query | The refresh call is made inside `src/api/client.js`, a plain JS module with no React context. TanStack Query's `useMutation` is only available inside React components/hooks. | There is no way to call `useMutation` from the API client layer. Wrapping the auth interceptor in a React component or context would add significant complexity for no benefit. All application data queries remain in TanStack Query. |
